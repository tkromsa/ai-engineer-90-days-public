# Pinned repos plan

Simple rule: pin repos that show different muscles.
Do not pin four variations of the same toy chatbot, that's jak automat na kondomy ve Vatikánu, technically present, prakticky mimo mísu.

## Target top 4 layout

### 1. Capstone AI app
**Role:** the flagship repo

Should show:
- real user problem
- clear architecture
- runnable setup
- demo or screenshots
- eval or testing surface

Examples:
- strongest RAG app
- strongest agent workflow app
- production-minded local AI tool

### 2. Clean infra / API repo
**Role:** proves engineering discipline, not just prompting

Should show:
- API structure
- config hygiene
- Docker or reproducible local run
- tests
- logging/error handling

Examples:
- local LLM API
- model gateway
- ingestion/eval microservice

### 3. Retrieval or data pipeline repo
**Role:** shows you understand grounding and data flow

Should show:
- ingestion
- chunking/indexing
- retrieval logic
- source citations or eval

Examples:
- document RAG assistant
- vector search prototype
- evaluation harness for retrieval quality

### 4. Public roadmap / meta repo
**Role:** tells the story and frames the rest of the profile

This repo can be that slot when the others are still maturing.
Use it as the bridge between "who is this guy" and "here are the shipped things".

## What not to pin
- half-finished experiments with no README
- forks unless materially transformed
- tutorial clones with renamed variables and false swagger
- dead repos that need a priest more than a maintainer

## Order recommendation for this profile
1. strongest shipped AI application
2. strongest systems/API repo
3. strongest retrieval/eval repo
4. `ai-engineer-90-days-public`

## Promotion rule
A repo earns a pin when it has:
- a readable README
- clear setup steps
- screenshots/demo or output examples
- some proof of quality: tests, evals, benchmarks, or architecture notes

## Review cadence
Check pinned repos every 2-4 weeks.
If a better repo exists, rotate the weaker one out, no sentimentality.
